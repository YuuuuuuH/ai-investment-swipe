
## 技术实现说明

### 筛选逻辑（经理为何不会看到重复项目）

`GET /api/feed` 查询 `projects` 表时，通过 `NOT EXISTS` 过滤掉已在 `human_feedback` 表中出现过、且版本号一致的记录，确保每位经理只看到未处理的最新项目。

### 数据导入与版本升级逻辑

`import_data.js` 解析每行的 `project_id` 与 `info_hash`，按以下三种情况处理：

| Case | 条件 | 行为 |
|------|------|------|
| Case 1 | 新 ID | 直接插入，`info_version` 设为 1 |
| Case 2 | 旧 ID + Hash 未变 | 重复信息，跳过 |
| Case 3 | 旧 ID + Hash 已变 | 项目内容更新，`info_version` 自增 +1 |

### 前端交互逻辑

- WXS 响应式滑动，用户松手后触发 `onCardSwipe` 发送请求
- 点击卡片切换 `isFlipped` 状态，翻到背面时请求 `/api/project/feedback/current`
- 卡片滑走后，在 `setTimeout` 里手动清空 `otherFeedbacks` 数组

### 管理员进度统计逻辑（`/api/admin/stats`）

后端遍历 `users` 表中的所有经理，对每位经理执行子查询：

- **分子**：该经理已反馈、且版本号等于项目表当前版本的记录数
- **分母**：项目表的总记录数

这样即使项目内容更新，进度环形图也能实时反映经理"针对最新信息"的处理进度。

## 后续优化空间

1. **虚拟列表渲染**：当前卡片全量加载渲染，数量增多后可能卡顿，需改为仅渲染序列前几张
2. **数据库索引**：现有查询未加索引，需优化以提升性能
3. **登录安全**：用户校验接口裸奔，且用户数据硬编码，正式上线前需加固
4. **决策意见交互**：底部输入框可改为抽屉式，或仅在滑向 Follow 时弹出提示填写意见
5. **历史看板 UI**：分类逻辑与多维度决策地图有待提升，便于展示决策分歧
6. **撤销功能**：支持返回上一张卡片，避免手滑误操作
